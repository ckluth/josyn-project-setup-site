ADR-0002: Tooling sandbox & reversibility

  • Date: 2026-07-17
  • Status: proposed
  • Promoted from: Idea 0006: Tooling sandbox & reversibility
  • Tags: tooling, policy, reversibility

Context

The project's "new way" runs on GitHub, deliberately free of policy-burden — a low-friction, experimental field that makes agentic ways of working feasible. That freedom is only acceptable to the organization because a switch back to the Azure DevOps corporate standard (work items, pipelines, standard policy) stays guaranteed. Reversibility is therefore a hard constraint, not an afterthought.

For that guarantee to hold, "reversible" needs a definition sharp enough to gate tooling choices rather than merely inspire them.

Portable — the term the constraint rests on — means: an artifact's authoritative content and history survive a move to Azure DevOps without loss of meaning, using only a mechanical (non-reconstructive) transformation. An asset is portable when:

  1. Its source of truth is medium-neutral — plain files in a git repository (decision-trail markdown, code, configuration), not locked inside a GitHub-only surface (Issues, Projects, Discussions, wiki-as-database).
  2. The switch-back is a re-mapping, not a rewrite — moving it to the Azure DevOps equivalent is a deterministic transform of existing content, not a "reconstruct the reasoning from memory" exercise.
  3. History travels with it — the decision trail (why / when) already lives in commits and markdown, independent of the hosting platform.

Decision Drivers

  • The low-policy sandbox is what makes agentic experimentation feasible.
  • The guaranteed return path is what makes the sandbox organizationally acceptable.
  • The switch-back must be low-effort — feasible on demand, not a heroic project.

Considered Options

  • A — All relevant content lives in the repos; tooling is not authoritative. Docs and code are the source of truth in git; any platform feature is a disposable view over them.
  • B — Use GitHub's proprietary surfaces (Issues, Projects) as the authoritative tracker, and rely on export/API migration if a switch-back is ever needed.
  • C — Soft preference for portability, allowing case-by-case exceptions where a proprietary feature is convenient.

Why not B. Making GitHub Issues + Projects the authoritative work-item tracker fails all three portability tests: the authoritative state (statuses, custom fields, board columns, comment history, automation) lives inside GitHub's database; the switch-back becomes a rewrite (custom-field semantics and automation have no deterministic 1:1 transform to Azure Boards); and issue timelines / label history never travel in commits. That is exactly the irreversible divergence the constraint exists to prevent.

Why not C. A soft preference cannot gate choices — under low effort and convenience pressure, exceptions accumulate until the return path quietly erodes. Reversibility only stays a guarantee if it is a hard line.

Proposed decision

Reversibility is a hard constraint. Concretely:

  1. All relevant content — docs and code — lives in the git repositories. The repos are the single source of truth. Anything that matters to the project's decisions, work, or execution is a file in a repo. Concretely, these are private repositories under the haevg-rz GitHub organization.
  2. No tooling-choice binding. No platform-proprietary surface may become the authoritative record of anything relevant. Platform features (GitHub Issues, Projects, etc.) are permitted only as disposable views over repo content, never as the source of truth.
  3. A clean, low-effort switch to Azure DevOps must always be possible, carrying all relevant content. Because the authoritative state is portable by construction, the switch-back stays a mechanical re-mapping (markdown and pipeline-as-code re-mapped to work items and Azure Pipelines), not a reconstruction.

Every tooling choice is tested against this line: if it were removed tomorrow, does all relevant content still survive in the repos and re-map mechanically to Azure DevOps? If not, the choice is not allowed to hold authoritative state.

Consequences

  • The decision-trail markdown under docs/ and the project's code are, by design, on the portable side of the line — this project already works this way.
  • Pipeline-as-code (.github/workflows/*.yml) is acceptable: proprietary syntax, but plain files in the repo that re-map mechanically to Azure Pipelines YAML.
  • GitHub Issues / Projects and similar surfaces may be used for convenience but must never be the authoritative record; their loss must cost nothing that isn't already in a repo.
  • This ADR decides the principle. The concrete Azure DevOps mapping (which repo artifacts become which work items / pipelines, and the switch-back runbook) is left to a follow-up decision and its plan.
  • The switch-back is planned as a milestone criterion, not now. Its trigger — at which milestone, or under which conditions, the switch becomes due — belongs to the project's phase/milestone framing in Idea 0009: Project progress, where it is surfaced deliberately and not planned at this time.
