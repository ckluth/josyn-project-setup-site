ADR-0010: Project progress

  • Date: 2026-07-18
  • Status: proposed
  • Promoted from: Idea 0009: Project progress
  • Tags: process, agile, milestones

Context

Project progress must be transparently defined — carried in explicit phases / milestones, agile in spirit: visible, incremental, and reviewable rather than an opaque march. Anyone should be able to see where we are, what a milestone requires, and what comes next. This is the progress-mechanics face of the true agile way of working set in ADR-0004: that ADR settled the day-to-day operating model (talking as the collaboration, a single curator on main, no silent acceptance); this one settles how the project's phasing is made legible and steerable.

Transparent phasing is what makes the experimental way of working legible to the organization and steerable by the team. Milestones are also the natural home for criteria that gate big moves — including the guaranteed switch-back to Azure DevOps from ADR-0002.

This ADR sets the progress mechanism, not a frozen list of phases: the concrete phase set stays adaptive by design (true agile), refined as the project moves.

Decision Drivers

  • Transparency — where we are, what a milestone needs, and what is next must be visible without insider knowledge.
  • Portability — phasing must ride on plain markdown in the repo, losing nothing on a switch-back (ADR-0002).
  • Lightweight, no external tracker — consistent with ADR-0004: main-only, no GitHub issues/projects, no sprint ceremony.
  • True agile — adaptive, incremental, feedback-driven; the phase set itself must stay refinable, not locked.
  • No silent milestone completion — declaring a milestone met must be as deliberate and attributable as accepting an ADR (ADR-0003).

Considered Options

  • A — Trail-native milestones. Milestones are decision-trail artifacts: each is an ADR defining it and its gating criteria, executed via plans, reviewed by talking, and surfaced in overview.md and the travel diary — all portable markdown.
  • B — External agile tooling. Run phases/sprints in Jira, Azure Boards, or GitHub Projects. Familiar, but it makes a proprietary surface authoritative — breaking ADR-0002 portability and ADR-0004's no-issues lightweight rule.
  • C — Informal / no explicit phasing. Let progress emerge without named milestones. Cheapest, but opaque — it defeats the very transparency this axis exists to provide.

Why A. It delivers transparent, steerable phasing using machinery the project already has (the trail), stays portable and lightweight, and keeps the phase set adaptive. B sacrifices portability for ceremony; C sacrifices the transparency goal outright.

Proposed decision

  1. Milestones are trail-native, portable markdown — no external tracker. A milestone is defined in the decision-trail as an ADR that states the milestone and its gating criteria; its execution rides on plans (Implements:); its live state is visible through docs/overview.md and narrated in the travel diary. Nothing authoritative lives in a proprietary surface (per ADR-0002 and ADR-0004), so phasing survives a switch-back intact.

  2. An initial, deliberately-adaptive phase sketch. As a starting frame — not locked, refinable at any time (true agile):

    • Phase 1 — Found the way of working: the method, human/culture, and governance ADRs, plus ratification of inherited work (ADR-0009).
    • Phase 2 — Technical re-decisions: the re-hosted josyn architecture threads carried through the collaborative trail.
    • Phase 3 — Delivery & switch-back readiness: clean target repos and the point at which the Azure DevOps switch-back becomes due.

    This sketch orients; it does not bind. Concrete milestones and their criteria are fixed by their own ADRs as the project reaches them.

  3. "Agile" here is pull- and feedback-driven, not calendar ceremony. Progress is made of coherent incremental trail moves (ADR-0004) and plan checkboxes; the cadence is pull-based, paced by feedback and readiness rather than fixed sprints — consistent with the pull-based adoption pace of ADR-0006. The review rhythm is continuous conversation plus a reviewable checkpoint at each milestone boundary. No cargo-cult scrum.

  4. The Azure DevOps switch-back is a milestone criterion — surfaced now, trigger deferred. It is owned in this trail as a milestone-gated event, but at which milestone (or under which conditions) it becomes due is deliberately not set now — it is fixed by a future milestone-defining ADR, matching the deferral already recorded in ADR-0002.

  5. Milestone criteria are defined, reviewed, and marked met without silence. Criteria are defined in the milestone ADR (the criteria are the decision), reviewed by talking, and accepted under ADR-0003 / ADR-0004 governance. Declaring a milestone met is a guarded, visible, attributable trail act — like ADR acceptance, a deliberate recorded move (date, that the team judged the criteria met, who) — never a silent flip. Execution progress shows in plan checkboxes; the met judgment is a recorded team call.

Consequences

  • Progress becomes legible and steerable from the repo alone — no insider knowledge, no external tool — at the cost of the discipline of writing each milestone and its criteria into the trail.
  • Because phasing is plain markdown, a switch-back per ADR-0002 loses no progress substance.
  • The pull-based, no-sprint cadence trades predictable calendar dates for genuine feedback-driven pacing — deliberate, consistent with ADR-0006.
  • No-silent-completion makes each milestone boundary a greppable, attributable moment, mirroring ADR-0004's no-silent-acceptance rule.
  • This ADR sets the mechanism and an adaptive sketch; the concrete milestones, their criteria, and the switch-back trigger are follow-up milestone ADRs and their plans, authored as the project reaches them.
